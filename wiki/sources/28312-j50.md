## Summary

3GPP TS 28.312 defines Intent Driven Management Services (IDMS) for mobile networks. It introduces the intent concept — expressing "what" needs to be achieved rather than "how" — and specifies concepts, use cases, requirements, Stage 2 information model (Intent IOC, IntentReport IOC, IntentHandlingFunction IOC, and numerous dataType definitions), Stage 3 RESTful HTTP solution sets, and scenario-specific intent expectations for radio network delivery, radio service, edge service, coverage assurance, RAN UE throughput, E2E optimization, RAN energy saving, 5GC network, and network maintenance.

## Key Claims

- Intent shifts management focus from "how" (rules, policies) to "what" (desired outcomes); intent is a higher abstraction than policy, expressing expectations without specifying implementation
- Three intent types by user role: Intent-CSC (Communication Service Customer), Intent-CSP (Communication Service Provider), Intent-NOP (Network Operator)
- Two categories of intent expectations: (1) delivering network/service objects, (2) assuring performance of existing objects
- Intent IOC contains IntentExpectation(s) — each expectation targets an ExpectationObject with ExpectationTarget(s) (targetName + condition + value range) and optional Context(s)
- Intent lifecycle states: ACKNOWLEDGED → COMPLIANT → FULFILLED → DEGRADED, with SUSPENDED and FULFILLMENT_FAILED as alternate paths; TERMINATED on deletion
- Intent conflict types: target conflict (within same expectation), expectation conflict (within same intent), intent conflict (between different intents)
- Intent negotiation in two phases: pre-evaluation (feasibility check, exploration) and fulfilment (negotiating alternatives, best outcomes, utility functions)
- IntentUtilityFunction allows consumers to express preference via mathematical functions (variables, weights, function type, result) for quantitative fulfilment evaluation
- IntentReport IOC aggregates six report types: fulfilment, conflict, feasibility check, exploration, fulfilment negotiation, utility
- IntentHandlingFunction IOC exposes producer capabilities (supported expectation objects, targets, value ranges)
- Scenario-specific expectations defined for: radio network delivery, radio service, edge service, coverage assurance, RAN UE throughput, RAN capacity, E2E optimization, RAN energy saving, 5GC network, network maintenance
- Stage 2 reuses generic provisioning MnS from TS 28.532; Stage 3 provides RESTful HTTP solution set with OpenAPI specs

## Key Quotes

> "An intent specifies the expectations including requirements, goals and constraints given to a 3GPP system, without specifying how to achieve them." — Section 3.1

> "An intent focuses more on describing the 'What' needs to be achieved but less on 'How' that outcomes should be achieved. The intent expresses the metrics that need to be achieved and not how to achieve them." — Section 4.1.1

> "An intent is typically understandable by humans, and also needs to be interpreted by the machine without any ambiguity." — Section 4.1.1

> "Intent can be used for management and control of closed-loop automation (e.g. intent can be used to specify the goals for the closed-loop), which means the intent can be translated to policies and management tasks." — Section 4.3

> "As it now stands, the telecom systems are mainly focused on 'how' and 'less what'... the need for an abstraction level description (i.e. Intent) becomes more apparent." — Section 4.4

## Contradictions

- TS 28.556 ([[PolicyIOC]]) uses a simple condition-action model for policy content, while TS 28.312 ([[IntentIOC]]) uses a richer ExpectationObject + ExpectationTarget + Context model for intent content. These are complementary rather than contradictory — intent is a higher abstraction that can be translated down to policies.
- TS 28.556 ([[PolicyConflictDetection]]) defines a single-level policy conflict notification, while TS 28.312 defines three granular levels of intent conflict (target, expectation, intent). The intent conflict model is more detailed.
